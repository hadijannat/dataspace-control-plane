# OpenAPI Reference

The control-api OpenAPI specification follows a three-directory structure that separates hand-authored source files from generated artifacts.

## Directory Structure

```
docs/api/openapi/
├── source/
│   └── control-api.yaml          # Source spec (hand-authored / FastAPI-generated)
├── bundled/
│   └── control-api.yaml          # Redocly-bundled single-file output
└── index.md                      # This file
```

### `source/`

The `source/` directory contains the design-time OpenAPI 3.1 specification. For the control-api, this is `control-api.yaml` — a single-file spec that documents the full API surface including all paths, request/response schemas, security schemes, and server declarations.

**Authoring policy**: The source spec is maintained by docs-lead and apps-lead jointly. When the FastAPI implementation diverges from the source spec, the implementation takes precedence — update `control-api.yaml` to match. The committed spec is a design contract; the running application is the authoritative implementation.

**Generating from FastAPI**: Once the control-api is running, the spec can be exported directly:

```bash
# Export from running control-api
curl http://localhost:8000/openapi.json | python3 -m json.tool > /tmp/openapi.json
# Convert to YAML and compare with source spec
```

Or via FastAPI's built-in export (requires the app to be importable):

```bash
python3 -c "
from apps.control_api.main import app
import yaml, json
print(yaml.dump(json.loads(app.openapi()), allow_unicode=True))
" > docs/api/openapi/source/control-api.yaml
```

### `bundled/`

The `bundled/` directory contains the Redocly-bundled single-file output. This is what is deployed to the documentation site and served via Redoc or Swagger UI.

**Do not edit `bundled/` by hand.** It is generated by the `redocly bundle` step. Edit the source spec, then re-bundle.

## Tooling Commands

### Lint the source spec

```bash
redocly lint docs/api/openapi/source/control-api.yaml
```

Linting checks: schema validity, broken `$ref` references, missing `operationId`, missing `description` fields, and security scheme consistency. Redocly CLI 2.x+ is required for OpenAPI 3.1.0 support.

Install Redocly CLI:

```bash
npm install -g @redocly/cli@latest
# or
npx @redocly/cli@latest lint docs/api/openapi/source/control-api.yaml
```

### Bundle the spec

```bash
redocly bundle docs/api/openapi/source/control-api.yaml \
  -o docs/api/openapi/bundled/control-api.yaml
```

### Build the reference docs (Redoc)

```bash
redocly build-docs docs/api/openapi/bundled/control-api.yaml \
  -o site/api-reference/index.html
```

## CI Integration

The following checks run in CI on every pull request that modifies files under `docs/api/openapi/` or `apps/control_api/`:

1. `redocly lint docs/api/openapi/source/control-api.yaml` — fails on any lint warning in `strict` mode
2. `redocly bundle` — fails if bundling produces errors
3. Schema diff check — compares bundled output against the previous tag using `diff_schema.py`; breaks if classified as `breaking` without a version bump
